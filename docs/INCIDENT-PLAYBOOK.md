# Incident Playbook

> MakeIT | Версия 1.0 | Март 2026

---

## 1. Обнаружение инцидента

**Источники алертов (в порядке приоритета):**

1. 🤖 Telegram от `@makeit_alert_bot` — автоматически при падении сервиса
2. 📧 Email от Better Stack — дублирующий канал
3. 👤 Сообщение от клиента

**При получении алерта → сразу открыть:**
- [MakeIT Dashboard](https://sergio1990-1.github.io/makeit-dashboard/) — общий статус
- [Better Stack](https://uptime.betterstack.com) — детали монитора

---

## 2. Классификация (5 минут)

```
Сервис полностью недоступен?
  ДА → P1-critical
  НЕТ → Ключевая функция сломана?
          ДА → P2-high
          НЕТ → Деградация/тормоза → P3-medium
```

---

## 3. Реакция по уровню

### P1 — Полное падение сервиса

**Шаги:**

```bash
# 1. Проверить статус контейнера
ssh deploy@89.167.17.79 "docker ps | grep <project>"

# 2. Посмотреть последние логи
ssh deploy@89.167.17.79 "docker logs <container> --tail 50"

# 3. Перезапустить если завис
ssh deploy@89.167.17.79 "docker restart <container>"

# 4. Проверить диск/память
ssh deploy@89.167.17.79 "df -h && free -h"
```

**Уведомить клиента (Tier 1) — шаблон:**
```
[Инцидент] <Название сервиса> недоступен

Зафиксировано падение сервиса в [время].
Работаем над восстановлением.
Следующий апдейт через 30 минут.
```

**Если не восстановилось за 30 мин → эскалация:**
- Проверить VPS: `ssh deploy@89.167.17.79 "uptime && dmesg | tail -20"`
- Проверить хостинг Hetzner на инциденты
- Рассмотреть откат к предыдущей версии

---

### P2 — Частичная деградация

1. Воспроизвести проблему локально
2. Найти в логах момент появления (`docker logs --since 1h`)
3. Создать GitHub Issue с лейблом `P2-high` + `bug`
4. Добавить в MakeIT Tracker: `gh project item-add 1 --owner Sergio1990-1 --url <url>`
5. Уведомить клиента если это Tier 1

---

### P3 — Деградация

1. Создать GitHub Issue с лейблом `P3-medium` + `bug`
2. Добавить в трекер
3. Запланировать в ближайший спринт

## 5. Восстановление

**После устранения инцидента:**

1. Убедиться что Better Stack показывает `up`
2. Убедиться что в Telegram пришёл `🟢 RECOVERED`
3. Уведомить клиента (Tier 1):

```
[Resolved] <Название сервиса> восстановлен

Сервис работает в штатном режиме с [время].
Длительность инцидента: [X] минут.
Причина: [кратко]
```

4. Создать GitHub Issue с лейблом `bug` для post-mortem (P1/P2)

---

## 6. Post-mortem (для P1)

Создать issue в репо проекта в течение 24 часов:

```markdown
## Post-mortem: [название инцидента]

**Дата:** [дата]
**Длительность:** [X] минут
**Affected:** [проект/клиент]

### Что случилось
[кратко]

### Причина
[root cause]

### Что сделано
[действия по устранению]

### Что изменим
- [ ] [предотвращающая мера 1]
- [ ] [предотвращающая мера 2]
```

---

## 7. Чеклист дежурного (еженедельно)

- [ ] Проверить Better Stack — нет ли длительных инцидентов
- [ ] Проверить `df -h` на VPS — диск не забит
- [ ] Просмотреть open P1/P2 issues в трекере
- [ ] Проверить Dependabot PR — смержить security updates
