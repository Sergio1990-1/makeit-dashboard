Процесс: от нового проекта до production (end-to-end)

PM (Я) управляет процессом. Разработчик (Я) использует Claude Code как инструмент (⚡). Pipeline и Auditor работают автоматически.

← Прокрутите диаграмму горизонтально →

Поток A: Новый проект (end-to-end)

  1. Клиент Обращение клиента — звонок, встреча, ТЗ
  2. Я Провожу встречу / созвон → делаю аудиозапись
  3. Dashboard Загружаю аудио в Транскрипты → intake → STT (Whisper) → enrichment → structuring → synthesis
  4. Dashboard Получаю BRIEF.md — структурированный документ с решениями и требованиями
  5. Claude Code Запускаю /makeit-init — создаётся repo, docs, CI, labels, .env, Dockerfile, pre-commit
  6. Claude Code Запускаю /makeit-plan с BRIEF → SPEC → PRD → Epic → GitHub Issues + Milestone с дедлайном
  7. Решение по каждому Issue: auto (Pipeline) или manual/assisted (Claude Code)?
  8. Pipeline Auto-задачи → queued → dev → self_check → PR → review → QA → merge (полностью автоматически)
  9. Claude Code Manual-задачи → /makeit-dev → код + тесты + PR
  10. Claude Code /makeit-codereview → ревью PR (diff-first, evidence-based)
  11. Ревью пройдено? Да → Merge. Нет → обратно на /makeit-dev
  12. Я Merge PR в main
  13. Dashboard Запускаю аудит проекта
  14. Auditor L1 (ruff, mypy, semgrep, eslint) → L2 (AST indexing) → L3 (LLM на GPU, Qwen3-Coder) → L4 (Reports)
  15. Dashboard Верификация findings → Claude читает код → CONFIRMED / FALSE_POSITIVE / UNCERTAIN
  16. Dashboard Создание GitHub Issues из верифицированных findings (1-5 issues по темам)
  17. Issues из аудита: auto → Pipeline, manual → Claude Code dev цикл
  18. Claude Code /makeit-deploy → pre-flight (9 проверок) → backup → deploy → health check → tag release
  19. Claude Code Если health check fail → автоматический rollback

Поток B: Обратная связь клиента

  1. Клиент Обратная связь: баг-репорт, запрос фичи, изменение требований
  2. Я Аудиозапись / текстовые заметки
  3. Dashboard Загрузка в Транскрипты → BRIEF.md
  4. Claude Code /makeit-plan → Issues (добавляются к существующему Milestone)
  5. → Подключение к основному dev-циклу (шаг 7 Потока A)

Поток C: Периодический аудит (quality gate)

  1. Dashboard Запуск аудита проекта
  2. Auditor L1→L2→L3→L4 → Findings
  3. Dashboard Верификация → Issues
  4. → Issues → Dev цикл (авто или ручной)

Поток D: Research & Discovery (планирование фич)

  1. Dashboard Запуск Research Agent (product_description, region)
  2. Pipeline Research Agent → RESEARCH.md (рынок, конкуренты)
  3. Dashboard Запуск Discovery Agent (на основе research)
  4. Pipeline Discovery Agent → DISCOVERY.md (gap matrix, фичи)
  5. → /makeit-plan → Issues → Dev цикл

Поток E: Debate (архитектурное решение)

  1. Dashboard Создание дебата: тема + позиции
  2. Pipeline Proposal → Critique ×N → Final Position → Synthesis (Opus)
  3. Я Опционально: вмешательство в раунде
  4. Pipeline → ADR (Architecture Decision Record)
  5. → Решение принято → может привести к /makeit-plan для новых задач

Система самообучения MakeIT

Замкнутый цикл: каждая задача → метрики → ретроспектива → предложения → автоприменение → мониторинг KPI → rollback при деградации.

Цикл самообучения

KPI метрики (что отслеживается)

success_rate

% задач успешно замерженных

first_pass_rate

% задач без retry

retry_rate

% задач с повторными попытками

avg_cost

Средняя стоимость задачи ($)

qa_pass_rate

% задач прошедших QA

finding_density

Среднее findings на задачу

Что обучается, а что НЕТ

✅ Обучается

• Pipeline code generation (dev agent quality)
• Pipeline rules (review/testing rules)
• Pre-triage config (skip keywords)
• Project-local lessons-learned

Работает

❌ НЕ обучается

• Auditor (false positive tuning passive)
• Транскрипты (нет feedback loop)
• /makeit-plan (нет метрик качества планов)
• Research/Discovery (нет оценки)
• Бизнес-процессы (нет BP Agent)

Gap-анализ: что отсутствует в процессах

Обнаруженные пробелы в текущей системе MakeIT — по результатам анализа кода и описания рабочего процесса.

🔴 Агент по бизнес-процессам

Сейчас нет агента, который по BRIEF/спецификации сам выстраивает цепочку бизнес-процессов. Человек (ты) вручную определяет порядок шагов: «сначала init, потом plan, потом dev...». Это знание живёт в голове, а не в системе.

Нужно: Агент, аналогичный Auditor, но для бизнес-процессов — проверяет, что все шаги учтены, что бизнес-логика не нарушена, что артефакты связаны.

GAP — критический

🔴 Скилл для встраивания фичи в процесс

Есть /makeit-plan (новая фича → Issues), но нет скилла, который анализирует СУЩЕСТВУЮЩИЕ бизнес-процессы проекта и показывает, как новая фича в них встраивается. Пример: если клиент просит «добавить возврат товара» — нужно видеть, как это влияет на заказ → оплату → доставку.

GAP — высокий

🟡 Нет автоматического триггера аудита

Аудит запускается вручную. Нет автоматического запуска после N мержей или по расписанию. Pipeline делает QA-verify для отдельных PR, но полный аудит проекта — только вручную.

GAP — средний

🟡 Разрыв между BRIEF и /makeit-init

BRIEF генерируется в Dashboard (браузер), а /makeit-init запускается в Claude Code (терминал). Нет автоматической передачи BRIEF в инициализацию — копируется вручную.

GAP — средний

🟡 Нет единого статуса «где проект сейчас»

Dashboard показывает Issues/PRs/Milestones, но нет высокоуровневого состояния: «проект на стадии планирования» → «в разработке» → «на аудите» → «готов к деплою». Нет workflow-engine.

GAP — средний

🟢 Knowledge используется пассивно

makeit-knowledge содержит правила и скиллы, но агенты читают их только при запуске. Нет механизма проактивного применения: «в этом проекте используется Decimal для денег — проверь, что новый код не использует float».

IDEA — улучшение

🟢 Нет связи Research → Plan

Research и Discovery генерируют RESEARCH.md / DISCOVERY.md, но нет автоматической конвертации их findings в /makeit-plan задачи. Результаты просматриваются вручную.

IDEA — улучшение

🟢 AutoTuner → только Pipeline rules

AutoTuner предлагает изменения в pipeline rules/prompts, но не анализирует бизнес-процессы. Ретроспективы фокусируются на код-качестве, а не на процессных bottlenecks.

IDEA — улучшение

Концепт: Business Process Agent

Агент, который отвечает за бизнес-процессы — как Auditor отвечает за код. Анализирует, валидирует, предлагает.

Зачем нужен?

Сейчас: ты описываешь бизнес-процесс словами → Claude пытается интерпретировать → результат непредсказуемый. Знание о процессах живёт в твоей голове.

С BP Agent: процессы описаны как данные (BPMN/YAML/DSL) → агент проверяет их, находит пробелы, предлагает оптимизации.

1. Формализация процессов

Каждый проект получает docs/PROCESSES.md или docs/processes/*.yaml — описание бизнес-процессов в структурированном виде:

process: order_fulfillment
  trigger: client.creates_order
  stages:
    - validate_order → check_inventory
    - check_inventory → [available: reserve, unavailable: notify_client]
    - reserve → payment
    - payment → [success: deliver, failure: cancel]
  invariants:
    - "order.total == sum(items.price * items.qty)"
    - "payment.amount == order.total"
  artifacts: [Order, Invoice, DeliveryNote]

2. BP Audit (аналог Code Audit)

Проверяет:

• Все ли пути имеют завершение (нет deadlocks)
• Все ли состояния достижимы
• Есть ли обработка ошибок на каждом шаге
• Не нарушаются ли инварианты
• Совпадает ли код с описанным процессом
• Есть ли у каждого артефакта creator и consumer

Как Auditor, но для процессов

3. BP from BRIEF (генерация)

По BRIEF.md агент автоматически:

• Определяет какие бизнес-процессы затронуты
• Строит flow: trigger → stages → artifacts → end
• Предлагает state machine (draft → active → completed → archived)
• Показывает impact на существующие процессы
• Генерирует PROCESSES.md

Новый скилл: /makeit-process

4. Интеграция в pipeline

Где BP Agent встраивается в текущий workflow:

После BRIEF → до /makeit-plan:
BRIEF → BP Agent → PROCESSES.md → /makeit-plan (учитывает процессы)

После Code Audit:
Code Audit findings → BP Agent проверяет: «этот баг нарушает бизнес-процесс X на шаге Y»

При добавлении фичи:
/makeit-plan → BP Agent: «эта фича влияет на процессы A, B, нужно обновить шаги 3, 5»

5. Визуализация

BP Agent генерирует интерактивные BPMN-диаграммы (как эта страница) автоматически из PROCESSES.md. Диаграммы обновляются при изменении процессов.

Dashboard получает новую вкладку «Процессы» → визуализация всех бизнес-процессов каждого проекта.

6. Проверка по запуску

Автоматический режим: после каждого /makeit-plan — BP Agent проверяет, что новые issues не нарушают существующие процессы.

Ручной режим: запуск из Dashboard или Claude Code: /makeit-bp-audit — полный аудит бизнес-процессов проекта.

Findings: аналогично Code Audit — severity (critical/high/medium), описание, рекомендация. Можно создать Issues.

Обновлённый workflow с BP Agent

  1. Я Встреча с клиентом → аудиозапись
  2. Dashboard Транскрипт → BRIEF.md
  3. Claude Code /makeit-init → проект инициализирован
  4. BP Agent /makeit-process → анализ BRIEF → PROCESSES.md (бизнес-процессы, state machines, инварианты)
  5. Claude Code /makeit-plan (с учётом PROCESSES.md) → SPEC → PRD → Issues
  6. Dev цикл: /makeit-dev → /makeit-codereview → Merge
  7. Auditor Code Audit → Findings
  8. BP Agent /makeit-bp-audit → проверка: код соответствует бизнес-процессам? инварианты не нарушены?
  9. Claude Code /makeit-deploy → production
  10. Клиент Фидбек → новый BRIEF → BP Agent анализирует impact на текущие процессы → /makeit-plan
